    命名规则

    共性规则

    ​ 标识符直观可以拼读
    ​ 标识符的长度 符合最小长度 和 最大信息 的原则
    ​ 命名规则尽量与所采用的操作系统或开发工具的风格保持一致
    ​ 程序中不要出现仅靠大小写区分的相似的标识符
    ​ 程序中不要出现的标识符完全相同的全局变量和全局变量
    ​ 变量名应当使用 ’名词‘ 或者 ‘形容词 + 名词“
    ​ 全局函数应当使用 动词 或者 动词+名词
    ​ 类成员函数 应当只使用 动词命名

    简单的Windows应用程序命名规则

    ​ 类名和函数名用大写字母开头的单词组合而成
    ​ 变量和参数用小写字母开头的单词组合而成
    ​ 常量全用大写的字母，用下划线分割单词
    ​ 静态变量加前缀 s_
    ​ 如果不得已需要全局变量，则使全局变量加前缀g_
    ​ 类的数据成员加前缀m_(member)

    常量

    const 与 #define的比较

    ​ 建议 c++程序中只使用const常量

    常量定义规则

    ​ 需要对外公开的常量放在头文件中，不需要对外公开的常量放在定义文件的头部

    类中常量

    ​ const数据成员只在某个对象生成期内是常量，而对于整个类而言是可可变的
    ​ 类可以创建多个对象，不同的对象其const数据成员的值可以不同
    ​ const 数据成员的初始化只能在类构造函数的初始化表中进行
    ​ 通过枚举常量，实现整个类中都有恒定的常量

    函数设计

    参数的规则

    ​ 参数的书写要完整
    ​ 参数命名要恰当，顺序要合理
    ​ 如果参数是指针，且仅作输入时，则应在类型面前加const，防止在函数体内被意外修改
    ​ 如果输入参数以值传递的方式传递对象，则宜改用“const &”方式来传递
    ​ 避免函数有太多的参数，参数个数尽量控制在5个以内
    ​ 尽量不要使用类型和数目不确定的参数

    返回值的规则

    ​ 不要省略返回值的类型
    ​ 函数名字与返回值类型在语义上不可冲突
    ​ 不要将正常值和错误标志混在一起返回。正常值用输出参数获得，而错误标志用return语句返回
    ​ 有时候函数原本不需要返回值，但为了增加灵活性如支持链式表达，可以附加返回值
    ​ 如果函数的返回值是一个对象，有些场合用‘引用传递’ 替换 ‘值传递’可以提高效率。而有些场合只能用 ’值传递‘ 而不能用 ’引用传递‘，否则会出错

    函数内部实现的规则

    ​ 在函数体的入口处，对参数的有效性进行检查
    ​ 1.正确使用 断言（assert）来防止此类错误
    ​ 在函数体的出口处，对return语句的正确性和效率进行检查
    ​ 1.return语句不可返回指向栈内存的 ’指针‘或者’引用‘
    ​ 2.要该清楚返回值就是是值，指针还是引用
    ​ 3.如果函数返回值是一个对象，要考虑return的语句的效率
    ​ 其他建议
    ​ 1.函数功能要单一，不要设计多用途的函数
    ​ 2.函数体的规模要小，尽量控制在50行代码之内
    ​ 3.尽量避免函数带有记忆功能。 相同的输入应当产生相同的输出
    ​ 4.不仅要检查输入参数的有效性，还要检查通过其他用途进入函数体内的变量的有效性，例如全局变量，文件句柄
    ​ 1.用于出错处理的返回值一定要清楚，让使用者不容易忽视或误解错误的情况
    ​ 使用断言
    ​ 程序一般分为Debug版本和Release版本
    ​ Debug版本用于内部调试
    ​ Release版本发行给用户使用
    ​ 断言assert是仅在Debug版本其作用的宏，它用于检查“不应该发生的”的发生的情况
    ​ 使用断言捕捉不应该发生的非法情况，不要混淆非法情况与错误情况区别
    ​ 引用与指针的比较
    ​ 引用被创建的同时必须被初始化
    ​ 不能有NULL引用，引用必须与合法的存储单元关联
    ​ 一旦引用被初始化，就不能改变引用的关系

    内存管理

    内存分配方式

    ​ 从静态存储区分配
    ​ 内存在程序编译的时候就已经分配好，这块内存在程序的整个运行期间都存在，如全局变量，static变量
    ​ 从栈上创建
    ​ 栈内存分配的运算内置于处理器的指令集中，效率很高，但是分配内存容量有限
    ​ 从堆上分配，称为动态内存分配

    常见的内存错误及其对策

    ​ 内存分配未成功，却使用它
    ​ 内存分配虽然成功，但是尚未初始化就引用它。
    ​ 内存分配成功并且已经初始化，但操作越过内存边界
    ​ 忘记了释放内存，造成内存泄露
    ​ 释放了内存却继续使用它
    ​ 程序中的对象调用关系过于复杂
    ​ 函数的return语句写错了
    ​ 使用free或delete释放了内存后，没有将指针设置NULL，导致产生’野指针‘
    ​ 规则
    ​ 用malloc或new申请内存后，应该立即检查指针是否为null。防止使用指针值为null的内存
    ​ 不要忘记为数据和动态内存赋初值。防止将未初始化的内存作为右值使用
    ​ 避免数组或指针的下标越界
    ​ 动态内存的申请与释放必须匹配，防止内存泄露
    ​ 用free或delete释放了内存之后，立即将指针设置为null，防止产生野指针

    指针与数组的对比

    ​ 修改内容
    ​ 内容复制与比较
    ​ 计算内存容量
    ​ 注意当数组作为函数的参数进行传递时，该数组自动退化了同类型的指针
    ​ 指针参数是否传递内存的

    free和delete

    ​ delete只是把指针所值的内存给释放掉，并没有把指针本身设置为NULL

    杜绝“野指针”

    ​ 野指针不是NULL指针，是指向“垃圾”内存的指针

    有了malloc/free为什么还要new/delete

    ​ malloc 与 free 是 C++/C 语言的标准库函数，new/delete 是 C++的运算符

    内存耗尽怎么办？

    ​ 判断指针是否为NULL，如果是则马上用 return 语句终止本函数
    ​ 判断指针是否为NULL， 如果是则马上用exit（1）终止整个程序的运行
